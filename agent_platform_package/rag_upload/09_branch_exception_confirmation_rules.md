# 分支与异常确认规则

本材料用于 RAG 检索，帮助 Agent 设计分支和异常时区分“客户已确认”“RAG 建议”“Agent 推断待确认”。未经客户确认的异常策略不能写成最终事实。

## 核心原则

1. 异常可以建议，但不能冒充客户已确认。
2. 影响业务结果、数据口径、是否继续执行的分支必须客户确认。
3. 技术稳定性类异常可以给默认建议，但仍要标注为建议。
4. 报告中每个异常或分支都应有来源标签。

## 来源标签

| 标签 | 含义 | 报告表达 |
|---|---|---|
| 客户已确认 | 客户在问答中明确选择或补充 | “已确认：...” |
| RAG 建议 | 来自能力库、场景库或通用经验 | “建议预设：...” |
| Agent 推断待确认 | Agent 根据上下文推断，但客户未确认 | “待确认建议：...” |
| 必须补充 | 不确认就无法进入搭建 | “搭建前必补：...” |

## 可默认建议的异常

这些异常通常可以作为建议预设，但仍需在报告中标注来源为“RAG 建议”。

| 异常 | 建议处理 |
|---|---|
| 页面加载超时 | 有限重试，失败后截图并记录 |
| 元素定位失败 | 截图、记录页面和步骤，通知人工 |
| 文件不存在 | 记录缺失路径，停止当前文件处理 |
| 写入失败 | 保留源数据，记录失败位置，通知人工 |
| 通知发送失败 | 记录失败，不影响已完成业务结果 |
| 日志写入失败 | 尝试备用位置或记录到本地文件 |

## 必须客户确认的分支

这些分支会影响业务结果，不能由 Agent 自行决定。

| 分支 | 必须确认的问题 |
|---|---|
| 部分平台失败是否继续 | 是否仍生成报表？是否只汇总成功部分？ |
| 数据为空如何处理 | 代表真实为 0，还是待确认？ |
| 低置信度分类如何处理 | 自动归类、进入待确认，还是停止？ |
| 结果不一致如何处理 | 自动覆盖、人工复核，还是回滚？ |
| 高风险提交是否自动执行 | 是否允许无人值守点击提交、付款、删除、发起拦截？ |
| 日期口径选择 | 使用自然日、平台统计日、前一日还是自定义周期？ |
| 覆盖原有数据 | 是否允许覆盖，是否需要备份？ |
| 人工接管对象 | 异常通知谁，多久内处理，是否继续后续流程？ |

## 常见异常分级

| 等级 | 含义 | 建议继续策略 |
|---|---|---|
| 阻塞 | 当前流程无法继续，且影响最终结果 | 停止当前步骤或整次运行，通知人工 |
| 需人工确认 | 机器人无法可靠判断，但可保留证据 | 暂停当前对象，进入人工复核 |
| 可恢复 | 可能是网络、加载或临时问题 | 有限重试，失败后升级 |
| 仅记录 | 不影响核心业务结果 | 写日志并继续 |

## 报告写法规范

正确写法：

- “建议预设：页面加载超时时，机器人可重试一次并截图；具体重试次数需实施时确认。”
- “待确认：如果部分平台采集失败，是否仍生成日报，需要业务方确认。”
- “已确认：低置信度邮件进入待人工确认类别。”

错误写法：

- “部分平台失败时只汇总成功平台。”但客户未确认。
- “页面超时自动重试 3 次。”但重试次数未确认。
- “数据为空按 0 写入。”但空值含义未确认。

## 提问触发逻辑

当出现以下信号时，Agent 应优先追问，而不是直接生成异常策略：

- 用户提到多个平台、店铺、账号、文件或数据源。
- 用户提到“日报”“汇总”“写入表格”等结果对准确性敏感的场景。
- 用户提到分类、判断、识别、语义、相似、是否同类。
- 用户没有说明失败时是否继续。
- 用户没有说明成功标准或验收方式。
- 用户没有说明人工接管人或通知渠道。
