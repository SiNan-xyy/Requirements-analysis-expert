# 验证码能力边界判断

## 核心原则

验证码不是天然不可做，也不是天然可以无人值守。它应被判断为一类“条件能力项”：

- 有适配指令、频率可控、客户接受费用和准确率限制、允许人工兜底时，可以继续评估 RPA。
- 验证码类型未知、出现频繁、没有适配指令、客户不接受人工兜底或付费识别时，应标记为高风险。
- 无适配指令、无人工兜底、又要求高频无人值守时，才应作为强阻塞或不建议直接 RPA。

Agent 在模块 3 做能力边界判断时，不得只因为出现“验证码”三个字就判定不能做。

## 常见验证码类型

| 类型 | 典型表现 | 判断重点 |
|---|---|---|
| 数英类型 | 输入图片中的数字、字母、中文或混合字符 | 是否有通用数英识别指令；费用和准确率是否可接受 |
| 滑块类型 | 拖动滑块到缺口或指定位置 | 是否有平台专用滑块、通用轨迹或 AI 操作屏幕能力 |
| 点选类型 | 按图片提示点击指定文字、物体或区域 | 是否有点选适配指令；是否需要定制识别 |
| 旋转类型 | 旋转图片到正确方向 | 是否有京东等平台专项旋转识别能力 |
| 轨迹类型 | 按要求完成滑动轨迹 | 是否有轨迹验证能力；是否容易被平台风控识别 |
| 空间推理类型 | 根据提示选择空间关系、物体位置 | 是否有空间推理类适配指令；准确率需单独确认 |
| 曲线滑块 | 非直线滑块或复杂轨迹 | 目前应谨慎，优先确认是否有专项方案或人工兜底 |
| 未知类型 | 客户只说“有验证码” | 必须追问截图、平台、出现频率和处理方式 |

## 已知适配方向

来自验证码指令汇总表的可用信息表明，多数常见类型已有适配方向：

- 数英类型：可考虑通用数英验证码识别。
- 滑块类型：可考虑京东旋转、京东滑动轨迹、通用轨迹、阿里滑块、抖音滑块等专项或通用能力。
- 点选类型：可考虑京东图片点选、Shopee 点选、腾讯六宫格、空间推理点选、Temu 空间物体选择等能力。
- 复杂或未覆盖类型：可尝试自定义 AI 操作屏幕或处理验证码的 AI 能力，但通常速度更慢、稳定性更需要验证。

报告中可以写“可参考验证码识别/滑块/点选相关能力”，不要在客户未确认前承诺一定可自动通过。

## 必问信息

当客户需求涉及登录、平台后台、批量采集、自动下单、自动写入或高频访问时，模块 3 应主动追问：

1. 是否会出现验证码、扫码、短信验证、设备确认或二次验证。
2. 验证码大概是什么类型，是否可以提供截图或描述。
3. 验证码出现频率：首次登录、偶尔出现、每天出现、每次操作都出现。
4. 是否允许人工兜底，例如首次登录人工处理、失败后通知人工确认。
5. 是否接受付费识别服务，以及是否接受一定误识别率。
6. 是否有客户授权使用自动化工具访问该平台。
7. 验证码失败后是否允许跳过当前账号、当前平台或当前数据源。

## 判断口径

| 情况 | 建议结论 |
|---|---|
| 首次登录偶发验证码，允许人工处理 | 可继续评估，标记为需要人工兜底 |
| 常见验证码类型，有适配指令，客户接受费用和准确率限制 | 条件适合，需做小样验证 |
| 验证码类型未知，但客户可提供截图和平台信息 | 暂不阻塞，先补充确认 |
| 高频验证码，且客户要求完全无人值守 | 高风险，需要进一步验证或改造登录方式 |
| 无适配指令、无人工兜底、又要求高频无人值守 | 不建议直接 RPA |
| 涉及绕过平台限制、无授权访问或违反平台规则 | 不建议实施，应改走合规接口或人工流程 |

## 报告表达建议

可以这样写：

- “该需求存在验证码风险，但验证码不是天然不可做；需要先确认验证码类型、出现频率、是否有适配指令，以及是否允许人工兜底。”
- “若验证码仅在首次登录或低频出现，可设计为人工处理后继续执行。”
- “若需要付费识别服务，应在开发前确认费用、准确率和失败兜底策略。”
- “若验证码高频出现且无法使用适配指令或人工兜底，则不建议直接做无人值守 RPA。”

不应这样写：

- “有验证码，所以不能做。”
- “验证码可以自动绕过。”
- “不需要确认费用、准确率和授权。”
