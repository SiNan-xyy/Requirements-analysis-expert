# 需求文档字段说明

本材料用于 RAG 检索，帮助 Agent 把澄清结果映射到需求文档或方案打包字段。字段命名和最终 JSON 结构以 Git Skill 中模块 6 的 schema 为准。

## 需求背景

说明客户为什么提出这个需求，包括当前人工处理方式、痛点、频次、耗时、错误风险或管理诉求。

背景不应替代业务目标。背景回答“为什么要做”，业务目标回答“要做成什么”。

## 业务目标

说明机器人需要帮助客户完成的业务结果。

好的业务目标应包含对象、动作和结果，例如：

- 自动汇总多个电商平台的每日经营数据，并写入腾讯文档日报。
- 根据飞书消息中的物流单号发起物流拦截，并回传处理结果。
- 根据邮件发件人、主题和规则将邮件分类到文件夹，并生成整理清单。

## 起点和终点

起点对应触发条件，说明流程什么时候开始。

终点对应完成条件，说明做到什么程度算完成。

常见起点包括：

- 每天固定时间。
- 新消息到达。
- 文件上传到指定文件夹。
- 人工点击启动。
- 系统状态变为待处理。

常见终点包括：

- 数据写入完成。
- 状态更新成功。
- 文件生成并发送。
- 通知发送完成。
- 异常清单生成并交给人工。

## 输入数据

说明机器人执行前需要读取或接收的数据。

需要关注：

- 数据来源是否固定。
- 字段是否完整。
- 格式是否稳定。
- 是否存在缺失、重复、异常值。
- 是否需要映射表或规则表。

## 应用系统

说明机器人需要操作的系统、平台、文件、消息工具或网页。

需要关注：

- 是否有稳定账号权限。
- 是否经常需要验证码、短信或设备确认。
- 页面入口是否稳定。
- 是否支持导入、导出、下载、上传、复制、粘贴。
- 是否涉及多个店铺、账号、组织或权限范围。

## 输出结果

说明机器人最终产出的内容。

常见输出包括：

- 系统状态更新。
- 表格写入。
- 文件生成。
- 消息通知。
- 异常清单。
- 处理日志。

输出结果必须可验证，不能只写“处理完成”。

## 规则和判断

说明机器人什么时候做、什么时候不做、如何分类、如何匹配、如何汇总、如何判断异常。

如果规则无法写清楚，应先输出前置治理建议，而不是直接设计 RPA 流程。

## 风险和前置条件

需要记录影响可落地性的风险，例如：

- 规则不清。
- 输入不稳定。
- 目标系统不可稳定操作。
- 登录或验证码频繁。
- 结果不可验证。
- 异常无法归类。
- 缺少权限、模板、字段映射或样例数据。

前置条件可以包括：

- 统一模板。
- 准备账号权限。
- 确认字段映射。
- 建立分类规则。
- 准备测试样本。

## 验收标准

验收标准应说明如何判断方案成功。

可以包括：

- 覆盖哪些输入场景。
- 成功率或准确率目标。
- 异常是否能记录和通知。
- 日志字段是否完整。
- 输出结果是否能被业务人员核对。

## 方案打包

最终交付应区分：

- 对客版 HTML：面向业务客户，解释需求边界、方案流程、风险、前置准备和验收方式。
- 对开发版 HTML：面向开发人员，展示字段事实、流程卡片、异常设计、能力映射和待确认项。
- 结构化 JSON：作为唯一事实源，区分已确认事实、推断建议、缺失必填项和冲突不确定项。

HTML 只负责展示，不应创造新事实。开发人员应以结构化 JSON 和 Git Skill schema 为准。
