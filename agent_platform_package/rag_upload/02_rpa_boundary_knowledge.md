# RPA 能力边界知识

本材料用于 RAG 检索，帮助 Agent 解释 RPA 适合与不适合的常见边界。最终判断以 Git Skill 中模块 3 的规则和 schema 为准。

## RPA 更适合的工作

RPA 更适合处理规则清楚、动作重复、输入稳定、平台可操作、结果可验证的工作。

典型特征包括：

- 输入来源固定，例如固定 Excel、系统消息、标准表单、固定字段。
- 判断规则可以写成条件，例如状态为“待处理”才执行，金额大于某阈值需要复核。
- 操作动作重复，例如查询、复制、填写、下载、上传、移动文件、发送通知。
- 平台入口和页面相对稳定，机器人可以点击、输入、读取、下载或上传。
- 成功和失败有明确验证方式，例如页面提示成功、状态更新、生成文件、写入记录。

一句话判断：能写成规则，能重复执行，能验证结果，就适合优先考虑 RPA。

## RPA 不适合直接做的工作

以下情况不适合直接进入 RPA 自动化：

- 关键判断依赖人工语义理解，例如“判断是不是同一种物料”“判断客户真实意图”。
- 缺少稳定业务规则，无法说明什么情况做、什么情况不做。
- 输入数据格式经常变化，字段不固定，来源不稳定。
- 目标系统页面变化频繁，入口不固定，或经常需要人工干预。
- 登录、验证码、设备确认、短信验证频繁出现，且没有稳定处理机制。
- 结果无法验证，只能说“看起来处理了”，没有明确成功标准。
- 异常类型开放且不可归类，机器人遇到情况后无法判断下一步。
- 需求频次低、耗时少，但开发和维护成本高，投入产出比不合理。

这些情况不一定代表永远不能自动化，但通常需要先做规则治理、数据治理、权限治理或流程治理。

## RPA 的责任边界

RPA 不负责“想明白规则”，RPA 负责“稳定执行规则”。

如果业务规则还停留在人工经验、主观判断或口头习惯层面，应先把规则显性化，再考虑自动化。

例如：

- 不适合直接做：不同供应商对同一物料叫法不同，需要机器人判断是否同义。
- 更合适的前置工作：先建立标准物料名称和供应商名称映射表。
- 适合后续 RPA：映射表明确后，机器人按规则查询、匹配、入库或生成异常清单。

## 七个能力边界维度

模块 3 可围绕七个维度判断：

1. `scenario_match`：场景是否属于 RPA 常见可自动化场景。
2. `instruction_support`：影刀指令能力是否能支撑主要动作。
3. `input_readiness`：输入数据是否稳定、完整、可读取。
4. `rule_readiness`：判断规则是否可写清楚。
5. `platform_operability`：目标平台是否可稳定操作。
6. `result_verifiability`：处理结果是否可验证。
7. `exception_containment`：异常是否可归类、可记录、可交给人工处理。

指令存在只能作为证据，不能单独得出“可以自动化”的结论。还需要结合输入、规则、平台、验证和异常边界一起判断。

## 前置治理不是最终否定

当需求不适合直接 RPA 时，Agent 应区分：

- 暂时不具备条件：需要补规则、补数据、补权限、补模板。
- 不建议直接 RPA：核心判断无法规则化，或风险明显高于收益。
- 条件适合：可以继续进入流程拆解。

对于暂时不具备条件的需求，应给出前置治理建议，而不是简单说“不能做”。
