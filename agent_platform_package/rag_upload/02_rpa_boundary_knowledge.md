# RPA 能力边界知识

本材料用于 RAG 检索，帮助 Agent 解释 RPA 适合与不适合的常见边界。流程判断以 Git Skill 中的规则为准。

## RPA 更适合的工作

RPA 更适合规则清楚、动作重复、输入稳定、平台可操作、结果可验证的工作。

典型特征包括：

- 输入来源固定，例如固定 Excel、系统消息、标准表单。
- 判断条件可以写成规则，例如状态为“待处理”才执行。
- 操作动作重复，例如查询、复制、填写、上传、下载。
- 系统页面或接口相对稳定，机器人可以点击、输入、读取。
- 执行结果有明确提示、状态、文件、记录或日志。

一句话判断：能写成规则、能重复执行、能验证结果，就适合优先考虑 RPA。

## RPA 不直接适合的工作

不是所有重复操作都适合直接做 RPA。当判断依赖人工语义理解、缺乏统一规则、流程变化频繁时，不应直接进入自动化开发。

常见风险包括：

- 名称不统一，需要人工判断是不是同一类事物。
- 业务规则说不清，依赖“看情况”。
- 输入格式经常变化，需要人工理解。
- 结果没有明确成功或失败标志。
- 页面频繁改版，入口或按钮不稳定。
- 经常出现验证码、人脸、手机二次确认。
- 异常情况过多且没有分类规则。
- 低频且强人工判断，自动化收益不明显。

## 预处理不是最终否决

当需求存在前置治理问题时，应该建议先做标准化、规则梳理或映射表建设。这不是最终否定 RPA，而是说明当前还不适合直接进入流程设计或开发。
