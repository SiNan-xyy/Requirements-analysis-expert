# 物流拦截案例

本材料用于 RAG 检索，帮助 Agent 理解“物流拦截”类需求如何从一句话变成可判断的 RPA 需求。

## 初始需求

客户可能只说：

“我想让机器人自动做物流拦截。”

这句话不能直接进入开发，因为还缺少业务边界和能力边界信息。

## 应优先澄清的问题

可以围绕以下问题澄清：

- 拦截对象是什么？订单号、物流单号还是包裹？
- 拦截触发来自哪里？飞书消息、Excel 清单、系统待办还是人工按钮？
- 物流单号或订单号是否固定格式，是否可能缺失？
- 机器人在哪个系统里发起拦截？商家后台、物流平台、ERP 还是第三方系统？
- 发起拦截后，什么结果算成功？页面提示、状态变更、返回编号还是消息通知？
- 拦截失败如何处理？记录原因、重试、通知人工还是终止？
- 是否存在验证码、登录失效、权限不足或人工二次确认？

## 澄清后的边界事实示例

一个较清晰的边界事实可以是：

- 业务目标：根据飞书机器人收到的物流单号，在影刀商家后台发起物流拦截。
- 触发条件：飞书机器人收到符合格式的物流单号消息。
- 输入数据：物流单号，以及必要时的订单号或店铺信息。
- 操作系统：飞书、影刀商家后台或对应物流拦截后台。
- 输出结果：拦截成功或失败的通知消息。
- 完成条件：后台返回明确结果，并将结果发送回飞书。

## RPA 预筛观察

该场景通常具备一定 RPA 适配性，因为：

- 输入可以来自固定消息或表格。
- 操作动作可能比较重复。
- 平台通常可以通过页面点击、查询、提交完成。
- 成功或失败结果一般可以从页面状态或提示中获取。

但仍需重点确认：

- 物流单号是否稳定可识别。
- 后台入口和页面是否稳定。
- 是否频繁出现验证码或登录确认。
- 拦截失败原因是否能归类。
- 是否有权限对所有店铺或平台发起拦截。

## 更成熟的输出方向

适合的方案表达不是“机器人自动拦截物流”，而是：

“机器人接收飞书消息中的物流单号，校验格式后进入指定后台查询包裹状态，满足可拦截条件时发起拦截，并将成功、失败或需人工处理的结果回传飞书，同时记录处理日志。”

这样的表达能清楚呈现输入、规则、系统、输出、异常和验收标准。
