# 需求分析培训摘要

本材料用于 RAG 检索，帮助 Agent 解释需求分析方法。流程控制、字段合同和模块流转以 Git Skill 中的规则为准。

## 核心目标

需求分析的目标，是把客户的一句话想法逐步澄清为可分析、可评估、可落地的需求。

在 RPA 场景中，重点不是一开始就追问按钮、页面、点击路径，而是先确认业务边界：

- 客户真正想完成什么业务结果？
- 机器人什么时候开始？
- 机器人需要哪些输入？
- 机器人在哪些系统里操作？
- 什么结果代表任务完成？
- 成功或失败如何验证？

只有边界清晰后，才适合进入 RPA 能力边界判断、流程拆解、异常设计和方案打包。

## 六个边界事实

模块 2 需要优先澄清六个边界事实：

1. 业务目标：机器人要帮助客户完成什么业务事项？
2. 触发条件：流程什么时候开始执行？
3. 完成条件：做到什么程度算结束？
4. 输入数据：机器人开始前需要哪些数据？
5. 操作系统：机器人需要在哪些系统、平台、文件或消息工具中操作？
6. 输出结果：什么结果能够证明任务完成？

这六个事实对应 `clarification_result.boundary_facts`，是后续模块判断和打包的基础。

## 分析顺序

建议顺序是：

1. 先问业务边界。
2. 再做 RPA 适配性预筛。
3. 再进入能力边界判断。
4. 再做流程拆解。
5. 再设计异常、人审和日志。
6. 最后打包成对客版和对开发版交付物。

边界不清时，不应直接设计流程。流程不清时，也不应生成最终需求文档或开发方案。Agent 应先帮助客户补齐必要事实，并显式列出不确定点。

## 面向客户的表达方式

优先使用选择题，降低客户回答成本。

每个选择题应尽量保留：

- “暂不确定”选项；
- “其他”选项，并通过始终可见的补充输入框承载补充说明；
- 必要时允许客户自由补充。

客户补充的自由文本要先吸收进当前状态，再判断是否需要继续追问。不要重复追问客户已经高置信度回答过的问题。

## 判断信息是否足够

模块 2 的目标是达到“可进入 RPA 能力边界判断”的程度，而不是直接达到“可开发”的程度。

当六个边界事实基本明确，并且预筛没有发现明显阻塞时，可以进入模块 3。

当缺少关键边界事实时，应输出缺口报告。

当前置治理阻塞存在时，例如命名不统一、规则不可描述、输入不稳定，应先输出治理建议，而不是进入流程设计。
