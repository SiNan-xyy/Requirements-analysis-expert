# 报告质量规则

本材料用于 RAG 检索，帮助 Agent 生成更有价值的对客方案报告和实施对齐报告。目标是让客户和业务实施人员能看出：前面问答花费的时间被充分转化为清晰、可追溯、可落地的交付物。

## 总体原则

1. 报告必须体现问答成果，而不是泛泛总结。
2. HTML 只展示事实源，不创造新事实。
3. 已确认事实、建议方案、缺失信息和不确定点必须分开。
4. 推断内容不能写成客户已确认。
5. 流程步骤必须有输入、动作、输出、前置条件和风险。
6. 实施对齐版必须展示可参考的影刀能力。
7. 对客版和实施对齐版必须使用同一事实源。

## 对客方案报告质量要求

对客版适合业务客户阅读，应回答：

- 这个需求要解决什么问题？
- 当前是否适合继续推进 RPA？
- 方案大概会怎样运行？
- 客户还需要补充或确认什么？
- 哪些风险会影响无人值守？
- 下一步应该做什么？

对客版不应：

- 展示大量英文枚举或机器字段。
- 展示复杂指令清单。
- 把待确认内容写成已确定。
- 给出过度承诺，例如“整个过程无需人工干预”，除非客户已确认所有前置条件。

## 实施对齐报告质量要求

实施对齐版适合业务实施人员阅读，应回答：

- 哪些信息已经确认，可以用于搭建？
- 哪些信息搭建前必须补齐？
- 每个流程步骤做什么？
- 每个步骤可参考哪些影刀能力？
- 分支和异常哪些已确认，哪些只是建议？
- 日志、通知、人工复核如何设计？
- 验收标准是什么？

实施对齐版不应：

- 使用大量未翻译英文状态。
- 只给 5 到 7 个大步骤而没有能力依据。
- 只列异常，不标注来源。
- 生成精确点击路径、选择器、等待秒数或指令参数。

## 流程卡片最低要求

每个流程步骤至少包含：

- 步骤名称。
- 业务目的。
- 输入信息。
- 处理动作。
- 输出结果。
- 可参考的影刀能力。
- 前置条件。
- 依赖步骤。
- 可能分支。
- 待确认事项。

如果某一步没有明确能力依据，应写：

“暂无明确能力依据，需实施人员结合实际环境确认。”

## 分支和异常最低要求

每个分支或异常至少包含：

- 场景。
- 触发条件。
- 建议处理方式。
- 是否影响最终结果。
- 是否需要人工介入。
- 来源标签：客户已确认、RAG 建议、Agent 推断待确认、搭建前必补。

未经客户确认时，应使用：

- “建议预设”
- “待确认”
- “需要业务方确认”

不能使用：

- “已确定”
- “必须这样处理”
- “客户要求”

## 影刀能力展示要求

实施对齐版必须在流程步骤中展示可参考的影刀能力。

能力展示应是业务友好的，例如：

- 打开网页。
- 等待页面加载。
- 读取网页文本。
- 读取表格数据。
- 写入指定行列。
- 循环处理多个平台。
- 条件判断。
- 发送通知。
- 记录日志和异常清单。

可以在括号中补充影刀指令方向，但不要写具体参数。

## 问答价值体现要求

报告应让用户看出前面回答的问题被用到了哪里。

建议展示：

- “已确认信息”区域。
- “搭建前必补信息”区域。
- “待确认建议”区域。
- “来源追溯”区域。

来源追溯可以按业务语言展示：

- 来自客户回答。
- 来自 RAG 能力材料。
- 来自 Agent 推断，需确认。
- 需实施环境验证。

## 常见质量问题

### 问题 1：对客版过度确定

错误：

“整个过程无需人工干预。”

更好：

“在登录方式、模板结构和异常处理策略确认后，可评估是否支持无人值守运行。”

### 问题 2：实施版英文过多

错误：

“developer_alignment_status: needs_confirmation”

更好：

“搭建准备状态：需补充确认”

### 问题 3：流程缺少能力依据

错误：

“写入腾讯文档。”

更好：

“写入腾讯文档：可参考打开网页、定位 Sheet、按指定行列写入数据、写入后读取校验等能力。需确认登录方式、Sheet 名称和写入位置。”

### 问题 4：异常像 Agent 自己编的

错误：

“部分平台失败时，只汇总成功平台。”

更好：

“待确认建议：如果部分平台失败，可只汇总成功平台并标注失败平台；是否接受该策略需业务方确认。”

## 报告生成前检查

生成 HTML 前，Agent 应检查：

1. 是否有唯一事实源。
2. 是否区分已确认、推断、缺失和不确定。
3. 是否所有流程步骤都有能力依据。
4. 是否所有异常都有来源标签。
5. 是否存在英文状态未转中文。
6. 是否存在未确认内容被写成确定事实。
7. 是否存在对客版和实施版事实不一致。
