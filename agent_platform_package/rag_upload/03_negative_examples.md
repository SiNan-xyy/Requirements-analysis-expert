# RPA 预筛反例材料

本材料用于 RAG 检索，帮助 Agent 在模块 2 和模块 3 中识别常见风险。反例用于提示风险，不应在没有确认的情况下直接否定客户需求。

## 1. 物料名称不统一，需要语义判断

表现：

- 不同供应商对同一物料使用不同名称。
- 系统内没有统一命名标准。
- 是否属于同一物料需要人工理解。

为什么不适合直接 RPA：

- 问题不在于有没有操作步骤，而在于关键判断无法规则化。
- 机器人容易误判，造成重复建档、错误归类或错误入库。

更合适的下一步：

- 先统一物料主数据命名规则。
- 建立供应商名称和标准物料名称的映射表。
- 规则明确后，再考虑 RPA 自动入库。

## 2. 缺少稳定业务规则

表现：

- 客户只能说“看情况处理”。
- 无法说明什么情况执行、什么情况跳过。
- 不同员工处理方式不同。

风险：

- RPA 无法代替业务人员临场判断。
- 自动化后容易把不一致的人工习惯放大。

建议：

- 先把规则写成条件表、判断树或示例库。
- 无法规则化的部分进入人工确认。

## 3. 输入格式不稳定

表现：

- 数据来自不同模板、不同截图、不同邮件正文。
- 字段名称、位置、格式经常变化。
- 部分字段缺失，没有补救规则。

风险：

- 读取失败率高。
- 后续判断和录入会被错误输入影响。

建议：

- 统一输入模板。
- 建立字段校验规则。
- 对缺失字段设置人工补录或异常清单。

## 4. 结果不可验证

表现：

- 客户只要求“处理一下”，没有明确成功标准。
- 系统没有状态更新、成功提示或可核对记录。
- 无法确认机器人处理是否正确完成。

风险：

- 自动化完成后无法验收。
- 出错后难以追踪责任和修复。

建议：

- 明确成功标志。
- 设置处理日志、状态回写或结果清单。

## 5. 系统页面或入口不稳定

表现：

- 平台页面经常改版。
- 登录入口、菜单位置、按钮名称经常变化。
- 同一任务在不同账号下页面差异大。

风险：

- 机器人维护成本高。
- 流程容易中断。

建议：

- 优先确认是否有接口、导出文件、固定后台路径或稳定操作入口。
- 对页面变化设置监控和人工接管策略。

## 6. 强验证频繁出现

表现：

- 经常需要验证码、短信验证、设备确认、人脸识别。
- 登录态容易失效。
- 客户无法提供稳定授权方式。

风险：

- 机器人无法稳定无人值守执行。
- 自动化过程频繁卡住。

建议：

- 先确认账号权限、登录策略、白名单、企业授权或人工接管机制。

## 7. 异常开放且不可归类

表现：

- 可能出现很多未知情况。
- 客户无法列出主要异常类型。
- 每次异常都需要人工重新判断。

风险：

- 异常处理无法设计。
- 自动化只能覆盖很小一段理想流程。

建议：

- 先收集历史异常样本。
- 将异常归类为可重试、可跳过、需补资料、需人工确认、需终止等类型。

## 8. 低频且强判断

表现：

- 任务发生频率低。
- 每次处理都需要大量人工判断。
- 自动化节省时间有限。

风险：

- 开发成本和维护成本可能高于收益。

建议：

- 优先用模板、清单、规则表改善人工处理效率。
- 只有当频次提升或规则明确后，再考虑 RPA。
