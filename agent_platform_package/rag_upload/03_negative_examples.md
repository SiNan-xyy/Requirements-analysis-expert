# RPA 预筛反例材料

本材料用于 RAG 检索，帮助 Agent 解释常见不适合直接 RPA 的情况。Agent 不应仅凭客户第一句话下结论，应通过追问确认。

## 1. 物料名称不统一，需要语义判断

表现：

- 不同供应商对同一物料叫法不同。
- 需要判断两个名称是不是同一种东西。
- 没有标准命名规则或映射表。

更合适的下一步：

- 先统一物料主数据命名规则。
- 建立供应商名称与标准物料名称的映射表。
- 规则明确后再考虑 RPA 自动入库。

## 2. 缺少稳定业务规则

表现：

- 客户描述中经常出现“看情况”“一般来说”“人工会判断”。
- 无法明确什么情况做、什么情况不做。
- 同一场景不同人处理方式不同。

更合适的下一步：

- 先整理判断规则。
- 把常见情况分类。
- 无法规则化的部分保留人工审核。

## 3. 输入格式不稳定

表现：

- 输入来自聊天、表格、邮件、截图等多种形式。
- 字段名称、顺序、格式经常变化。
- 缺少固定模板。

更合适的下一步：

- 制定统一输入模板。
- 固定字段名称和必填项。
- 非标准输入先由人工预处理。

## 4. 结果不可验证

表现：

- 完成后没有页面提示。
- 没有状态变化、文件、记录或日志。
- 只能靠人工感觉判断是否成功。

更合适的下一步：

- 明确成功标志。
- 明确失败标志。
- 增加日志或结果记录。

## 5. 系统页面不稳定

表现：

- 页面经常改版。
- 按钮位置或名称频繁变化。
- 入口路径不固定。

更合适的下一步：

- 确认系统近期升级计划。
- 优先寻找 API 或数据接口。
- 页面稳定后再开发 RPA。

## 6. 强验证频繁出现

表现：

- 每次登录都需要手机验证码。
- 经常出现滑块、图形验证码或人脸识别。
- 需要人工在手机 App 上确认。

更合适的下一步：

- 申请机器人专用账号。
- 确认是否可以保持登录态。
- 改用接口或授权方式。
- 保留人工验证节点，不做全自动。

## 7. 异常开放且没有分类

表现：

- 失败原因很多。
- 不同失败原因处理方式不同。
- 目前没有异常分类。

更合适的下一步：

- 整理常见异常类型。
- 明确每类异常的处理动作。
- 无法覆盖的异常统一转人工。

## 8. 低频且强判断

表现：

- 处理频率低。
- 每次处理都需要人工判断。
- 自动化后仍需要大量人工确认。

更合适的下一步：

- 先评估处理频率和耗时。
- 优先自动化高频固定动作。
- 低频复杂判断保留人工。
