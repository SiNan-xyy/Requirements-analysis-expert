# RAG 物料人工审查表 v1

生成日期：2026-06-29

本文用于人工审查 Agent 的 RAG 物料是否足以支撑“模糊需求到可落地 RPA 方案”的问答、判断、流程拆解、异常设计和报告生成。本文只做物料治理建议，不改变现有 Agent 逻辑。

## 审查结论摘要

当前可上传 RAG 包已从 5 份方法论材料扩展为 11 份中文材料，新增内容用于支撑影刀能力推荐、腾讯文档/在线表格、报表采集场景、分支异常确认、HTML 中文展示和报告质量控制。

原始 Excel/Markdown 源文件仍不建议直接上传；本次已先抽取高优先级能力和报告规则，整理成业务专员可读、Agent 易检索、报告可引用的中文 RAG。

优先补充方向：

1. 已建立影刀常用能力中文卡片库。
2. 已建立腾讯文档/在线表格专项能力库。
3. 已建立报表采集与日报生成场景 RAG，电商日报只作为样例。
4. 已建立分支与异常确认规则库。
5. 已建立 HTML 展示中文词典。
6. 已建立报告质量规则库。

## 审查口径

| 审查维度 | 说明 |
|---|---|
| 可读性 | 是否为可读中文，是否适合业务专员理解 |
| 可检索性 | Agent 是否容易从材料中检索到具体场景、能力和限制 |
| 可引用性 | 是否可以直接进入报告，还是只能作为内部判断依据 |
| 事实等级 | 是否区分客户已确认、RAG 支撑、Agent 推断、待客户确认 |
| 模块适配 | 支撑模块 2/3/4/5/6 中的哪些环节 |
| 风险边界 | 是否说明不适用场景、前置条件、异常和人工确认点 |

## 已补充 RAG 包

### 1. 影刀常用能力中文卡片库

状态：已新增 `06_yingdao_common_capability_cards.md`。

目标：把影刀指令从“技术指令名”转成“业务专员能理解的能力”。

建议字段：

| 字段 | 说明 |
|---|---|
| 能力名称 | 业务化中文名称，例如“打开网页”“读取网页文本”“写入指定单元格” |
| 对应影刀指令 | 可参考的影刀指令名称，可多个 |
| 适用场景 | 哪类业务需求适用 |
| 需要客户确认 | 使用前必须确认的信息 |
| 常见风险 | 失败原因或不稳定来源 |
| 报告展示方式 | 对客版是否展示、实施版如何展示 |

首批建议覆盖：

- 打开网页
- 等待网页元素出现
- 点击网页元素
- 填写网页输入框
- 获取网页元素文本
- 获取表格数据
- 读取 Excel/在线表格
- 写入 Excel/在线表格
- 按行循环处理
- 条件判断
- 发送飞书/钉钉/企业微信通知
- 记录日志和异常清单

### 2. 腾讯文档/在线表格专项能力库

状态：已新增 `07_online_spreadsheet_capabilities.md`。

目标：解决报告中“腾讯文档可以怎么做”表达不足的问题。

建议内容：

- 打开腾讯文档或在线表格。
- 定位目标 Sheet。
- 读取指定行、列、单元格或表格区域。
- 按指定行列写入数据。
- 判断写入是否成功。
- 处理权限不足、扫码登录、模板结构变化、单元格坐标变化。
- 区分“已确认可密码登录”和“仍需确认登录方式”。

必须人工确认的问题：

- 腾讯文档是否支持稳定登录。
- 是否需要扫码或二次验证。
- 目标 Sheet 名称是否固定。
- 写入行列或单元格坐标是否固定。
- 日期对应的写入行如何确定。
- 写入后是否需要自动核对源值与目标值。

### 3. 报表采集与日报生成场景 RAG

状态：已新增 `08_report_collection_and_daily_report_scenario.md`。

目标：覆盖“从一个或多个系统采集数据，并生成日报、周报、月报或运营看板”的高频需求。电商日报只是该类场景的一个样例，不单独作为长期独立 RAG。

适用场景示例：

- 电商平台日报。
- 广告投放日报。
- 财务日报。
- 运营数据日报。
- 多系统数据汇总。
- 销售日报。
- 库存日报。

建议流程卡：

1. 加载报表配置。
2. 登录目标系统或打开数据源。
3. 进入报表、数据看板或文件页面。
4. 选择日期口径。
5. 读取指标数据，例如 GMV、订单量、消耗、收入、库存、退款等。
6. 按平台、店铺、部门、项目或其他维度汇总。
7. 写入目标报表，例如腾讯文档、在线表格、Excel 或系统看板。
8. 记录日志。
9. 发送异常或完成通知。

必须确认的问题：

- 数据源清单是否固定，例如平台、店铺、账号、系统、文件或报表。
- 每个数据源的指标名称和统计口径是否一致。
- 报表日期是自然日、业务统计日、平台统计日还是前一日。
- 是否包含扣减项或补充项，例如退款、售后、取消订单、广告返点、库存调整。
- 部分数据源失败时是否仍生成报表。
- 是否要求自动核对源数据和目标报表写入值。
- 目标报表的 Sheet、行列、单元格或模板结构是否固定。

### 4. 分支与异常确认规则库

状态：已新增 `09_branch_exception_confirmation_rules.md`。

目标：防止 Agent 把自己推断的异常写成客户已确认事实。

建议分级：

| 类型 | 可默认建议 | 必须客户确认 |
|---|---|---|
| 登录失败 | 可建议记录并通知 | 账号权限、验证码、扫码、二次验证 |
| 页面超时 | 可建议重试和截图 | 最大重试次数、是否跳过 |
| 数据为空 | 可建议标记待确认 | 是否代表真实为 0 |
| 部分平台失败 | 可建议隔离处理 | 是否仍汇总成功平台 |
| 写入失败 | 可建议保留源数据并通知 | 是否允许人工补录 |
| 结果不一致 | 可建议人工复核 | 是否需要自动核对、核对口径 |

### 6. 报告质量规则库

状态：已新增 `11_report_quality_rules.md`。

目标：确保用户多轮问答的投入能在报告中体现。

建议规则：

- 报告必须展示“已确认信息来自哪些问答”。
- 每个流程步骤必须展示推荐影刀能力或说明暂无明确能力。
- 每个异常必须标注来源：客户确认、RAG 建议、Agent 推断待确认。
- 对客版避免英文枚举和技术字段。
- 实施版也应中文优先，英文仅保留在隐藏结构化 JSON。
- 不得把待确认项写成确定事实。
- 不得只输出大步骤，必须说明输入、动作、输出、前置条件和风险。
