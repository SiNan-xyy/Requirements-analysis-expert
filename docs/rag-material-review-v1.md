## 本轮新增：验证码能力边界

新增文件：`agent_platform_package/rag_upload/12_captcha_capability_boundary.md`

来源参考：`验证码问题及对应指令汇总表.xlsx`

新增原因：

- 历史规则容易把“有验证码”直接理解为 RPA 阻塞项。
- 影刀目前存在多类验证码适配指令或替代处理方式，不能简单判断为不可做。
- 验证码真正影响的是稳定性、成本、准确率、人工兜底、授权合规和无人值守程度。

判断原则：

- 验证码不是天然不可做。
- 有适配指令、频率可控、客户接受费用和准确率限制、允许人工兜底时，可以继续评估 RPA。
- 验证码类型未知时，应追问截图、平台、出现频率和处理方式。
- 无适配指令、无人工兜底、又要求高频无人值守时，才作为强阻塞或不建议直接 RPA。

支撑模块：

- 模块 3：判断验证码是否构成能力边界风险。
- 模块 4：在流程拆解中保留登录、验证码处理或人工兜底步骤。
- 模块 5：设计验证码失败、识别低置信度、触发平台风控等异常。
- 模块 6：在统一报告中解释验证码判断依据和待确认项。

## 审查口径

| 审查维度 | 说明 |
|---|---|
| 可读性 | 是否为可读中文，是否适合业务专员理解 |
| 可检索性 | Agent 是否容易检索到场景、能力、限制和反例 |
| 可引用性 | 是否可以进入报告，或只能作为内部判断依据 |
| 事实等级 | 是否区分客户已确认、RAG 支撑、Agent 推断和待确认 |
| 模块适配 | 支撑模块 2/3/4/5/6 中的哪些环节 |
| 风险边界 | 是否说明不适用场景、前置条件、异常和人工确认点 |

## 后续补充方向

1. 继续从影刀指令库抽取高频能力，但以“业务可读能力卡”形式沉淀，不直接上传大表。
2. 把更多行业场景拆成可复用能力卡、字段映射卡、异常卡，而不是每个场景单独做一份 RAG。
3. 持续补充“反例触发条件 + 提问策略”，减少 Agent 自行推断。
4. 将报告中所有面向客户和业务专员的字段保持中文优先。
