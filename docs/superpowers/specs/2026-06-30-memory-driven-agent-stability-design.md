# 需求记忆体驱动的 Agent 稳定性设计

## 背景

当前 RPA 需求分析 Agent 已具备模块 1 到模块 6 的主链路，也已支持验证码条件能力判断、模块自动流转和统一中文报告。但从真实测试看，仍存在一类核心稳定性问题：

- 模块流转依赖模型稳定输出标准字段。
- 长上下文、多轮问答、模型差异会导致事实丢失、重复提问或推断污染。
- 同一 Agent 对不同需求生成的报告结构相似但不完全稳定。
- GLM 等模型报告生成能力较稳定，但题型选择和事实来源约束仍不稳定。

因此需要从“上下文驱动 + 严格模块入参”升级为“需求记忆体驱动 + 事实槽位归档 + 就绪度门控”。

## 目标

1. 每个需求维护一份轻量“需求记忆体”，作为当前需求的唯一事实状态来源。
2. 每轮交互先读记忆体，再吸收用户最新回答，最后更新记忆体。
3. 模块流转不再依赖模型完美输出标准 JSON，而依赖事实槽位和就绪度门槛。
4. 支持 `ready`、`partial_ready`、`blocked` 三态流转，允许带缺口进入下一模块。
5. 最终报告必须引用记忆体中的事实编号，避免把推断写成客户确认。
6. 题型选择稳定支持单选、多选和补充项。
7. 统一报告结构稳定，不因需求场景变化而漏掉关键章节。

## 非目标

- 不把完整对话逐字保存为主记忆。
- 不要求模型每轮都完美输出严格 JSON。
- 不要求所有缺口都补齐后才能进入下一模块。
- 不用 RAG 保存单个客户需求状态；RAG 只保存通用知识。

## 核心架构

新的运行流程：

```text
用户输入
→ 读取需求记忆体
→ 抽取本轮新增/修正信息
→ 更新事实槽位
→ 判定模块就绪度
→ 进入下一模块、带缺口流转、回退补问或停止
→ 报告引用事实编号
```

需求记忆体是当前需求的事实源。上下文只作为辅助，不能覆盖记忆体中的已确认事实。

## 需求记忆体结构

建议文件名：

```text
requirement_memory.md
```

建议结构：

```markdown
# 需求记忆体

## 当前阶段
当前模块：module_3_rpa_boundary_check
当前状态：partial_ready
下一步：进入流程拆解，但保留 GMV 口径和验证码小样验证缺口

## 已确认事实 F
| ID | 主题 | 内容 | 来源 | 置信度 | 可用于开发 |
|---|---|---|---|---|---|
| F001 | 业务目标 | 自动统计电商平台日报并写入腾讯文档 | 客户回答 | 高 | 是 |

## 推断建议 I
| ID | 内容 | 依据 | 状态 | 可用于开发 |
|---|---|---|---|---|
| I001 | 建议先做单平台 PoC | 多平台登录和验证码风险较高 | 待确认 | 否 |

## 待确认缺口 G
| ID | 问题 | 阻塞级别 | 阻塞阶段 | 负责人 |
|---|---|---|---|---|
| G001 | GMV 口径是否各平台一致 | 高 | 开发前 | 客户 |

## 冲突或更正 C
| ID | 说明 | 处理方式 |
|---|---|---|
| C001 | 用户更正平台不包含抖音 | 移除抖音相关事实 |

## 已废弃问题 R
| ID | 内容 | 废弃原因 |
|---|---|---|
| R001 | 是否需要生成日报 | 用户已明确要写入日报表 |

## 模块就绪度 Gate
| 模块 | 状态 | 进入下一模块条件 | 当前结论 |
|---|---|---|---|
| module_2_requirement_clarification | ready | 业务目标、平台、输入、输出基本明确 | 可进入模块 3 |
```

## 事实编号规则

- `Fxxx`: 客户已确认或上游模块明确确认的事实。
- `Ixxx`: Agent 推断建议或 RAG 建议，必须待确认。
- `Gxxx`: 缺口、待确认问题或开发前必补项。
- `Cxxx`: 冲突、用户改口或事实更正。
- `Rxxx`: 已解决、已废弃或不应重复追问的问题。
- `Dxxx`: 模块级判断结论，例如 RPA 适配结论。

任何报告中的“客户已确认”只能引用 `Fxxx`。

## 模块流转门槛

### 模块 2 到模块 3

最低要求：

- 业务目标明确。
- 操作系统或平台明确。
- 输入数据或数据来源明确。
- 输出结果明确。

可暂缺：

- 触发条件。
- 完成条件。
- 字段映射。
- 详细异常策略。

### 模块 3 到模块 4

最低要求：

- RPA 分类不是明确不适合。
- 已有平台可操作性判断。
- 已有规则清晰度判断。
- 已有结果验证或待确认结果验证缺口。
- 已有主要风险。

### 模块 4 到模块 5

最低要求：

- 主流程可拆成 4 到 8 个业务步骤。
- 每步具备输入、动作、输出。
- 每步具备候选影刀能力或说明暂无明确能力。
- 至少标出需要异常设计的步骤。

### 模块 5 到模块 6

最低要求：

- 有关键异常。
- 有人工介入策略。
- 有日志或通知策略。
- 未确认异常必须标为待确认。

## 三态流转

- `ready`: 当前模块满足进入下一模块的必要事实。
- `partial_ready`: 当前模块可进入下一模块，但存在不阻塞下一阶段分析的缺口。
- `blocked`: 当前模块缺少关键事实，继续流转会导致无效分析或明显幻觉。

`partial_ready` 是默认推荐策略。Agent 应允许带缺口流转，并在后续报告中保留缺口。

## 题型稳定规则

题型选择不再交给模型自由判断，而使用业务规则：

- 互斥问题：单选 + 补充。
- 可并存问题：多选 + 补充。
- 涉及平台、数据来源、输出字段、异常处理、通知方式、人工兜底、验证码处理方式：优先多选 + 补充。
- 每题必须保留“其他，请补充”或等价选项。
- 不得把天然多选问题压成单选。

## 报告稳定规则

统一 HTML 报告固定包含：

1. 总体结论。
2. 需求理解。
3. RPA 适配判断。
4. 业务范围。
5. 流程步骤与候选影刀能力。
6. 异常处理与来源标签。
7. 事实分层。
8. 搭建前必补信息。
9. PoC 或下一步建议。
10. 中文化开发对齐 JSON 摘要。

不同需求可以内容不同，但章节不可随意缺失。

## 验收标准

1. 项目中新增需求记忆体 schema、模板和更新规则。
2. 模块 2 到模块 6 的流转规则改为读取事实槽位和 Gate 状态。
3. 题型规则明确何时必须多选。
4. 报告 schema 和规则要求固定章节、事实编号、异常来源标签和中文化开发对齐 JSON 摘要。
5. 新增稳定性测试覆盖电商日报、库存监控、邮件分类、物流拦截、物料同义判断、带验证码平台采集。
6. 测试不只检查 JSON 合法性，还检查报告结构稳定性和事实来源约束。
