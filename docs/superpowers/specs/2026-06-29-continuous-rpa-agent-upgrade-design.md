# 连续式 RPA 需求分析 Agent 升级设计

## 背景

当前 Agent 已完成模块 1 到模块 6 的基础链路：交互题型、需求澄清、RPA 边界判断、流程拆解、异常设计、方案打包。近期测试暴露出四类问题：

1. RPA 能力边界对验证码场景判断过粗，容易把“存在验证码”误判为天然不可做。
2. 模块 2 之后偏向 Agent 自行推理，缺少模块级追问，导致报告中部分异常和分支没有客户确认来源。
3. 模块之间跳转依赖用户手动发送“继续”，体验割裂。
4. 对客报告和开发报告拆分后，内容分散；业务专员既是提需人也可能是开发对齐人，更需要一份统一、中文、少 IT 黑话的落地分析报告。

## 产品定位

该 Agent 不与“一句话直接生成完整报告”的竞品做同质化竞争。它的核心定位是：

- 通过连续问答把模糊需求变成可判断、可拆解、可落地的 RPA 需求。
- 让客户参与关键判断，减少 Agent 自行脑补。
- 把 RPA 能力边界、指令可用性、前置治理、异常兜底讲清楚。
- 输出业务专员能看懂、开发人员也能执行的一份统一中文报告。

因此 Agent 应支持两种使用方式：

- 快速预评估：用户只想知道大概能不能做时，输出轻量边界结论和需要补充的信息。
- 落地分析：用户愿意参与多轮问答时，逐模块确认事实、风险、流程、异常和交付方案。

## 目标

1. 新增验证码能力边界 RAG，把验证码从“阻塞项”升级为“条件能力判断项”。
2. 模块 2 到模块 5 都具备各自的追问职责，问题必须来自当前模块缺口，而不是重复模块 1。
3. 支持模块自动流转：当上一模块达到进入下一模块的条件时，Agent 应说明进入原因并直接提出下一模块问题。
4. 模块 6 默认输出一份统一 HTML 报告，保留结构化事实源；如平台兼容需要，可保留旧版对客/开发视图作为兼容字段。
5. 所有面向客户、开发者、平台配置的材料默认中文表达，减少英文键名和专业 IT 术语暴露。

## 非目标

- 不把 Agent 改成一次性完整项目生成器。
- 不承诺验证码、登录风控、第三方平台限制一定可自动绕过。
- 不把所有行业场景逐个硬编码成独立 RAG；场景 RAG 只用于提炼可复用的能力卡、反例卡、字段映射卡和异常卡。

## 核心设计

### 验证码边界

验证码不再被视为天然不适合 RPA。模块 3 应按以下顺序判断：

1. 验证码类型：数英、滑块、点选、旋转、轨迹、空间推理、曲线滑块、未知。
2. 是否存在已知适配指令。
3. 是否需要付费识别服务，以及客户是否接受费用和准确率限制。
4. 是否允许人工兜底，例如首次登录、低频验证、失败转人工。
5. 是否满足客户授权、平台规则和合规要求。
6. 验证码出现频率是否会破坏无人值守稳定性。

只有在“无适配指令、无法人工兜底、又要求高频无人值守”的组合下，才应作为强阻塞或不建议直接做 RPA。

### 模块级追问

模块 1 负责统一题型和交互状态。

模块 2 追问需求边界：目标、触发、输入、系统、输出、完成条件、是否达到边界判断所需的最低信息。

模块 3 追问能力边界：平台可操作性、登录/验证码、数据获取方式、规则清晰度、结果验证、前置治理。

模块 4 追问流程拆解：执行入口、页面/文件/表格路径、读取字段、写入位置、循环对象、分支条件、指令候选。

模块 5 追问异常处理：哪些异常已由客户确认，哪些来自 RAG 建议，哪些是 Agent 推断且必须待确认。

模块 6 不再发散追问，只负责打包；若发现必需事实缺失，应回退到对应模块补问。

### 自动流转

每个模块都应输出：

- 当前模块结论。
- 已确认事实。
- 未确认但不阻塞的信息。
- 阻塞进入下一模块的信息。
- `next_module` 和进入原因。

若没有阻塞信息，Agent 应自动进入下一模块并直接提出下一模块的选择题，不要求用户手动发送“继续”。

### 统一报告

模块 6 默认输出一份《RPA 单需求落地分析报告》，包含：

1. 需求概述。
2. 是否建议做 RPA 及原因。
3. 实施范围和不做范围。
4. 主流程卡片。
5. 分支流程卡片。
6. 异常处理卡片，并标明信息来源：客户已确认、RAG 建议、Agent 推断待确认、开发前必须确认。
7. 指令/能力推荐，尤其是页面、表格、在线文档、验证码、登录等能力。
8. 前置准备清单。
9. 开发对齐 JSON。
10. 待确认问题。

统一报告应照顾业务专员阅读，字段名和标题使用中文。结构化 JSON 可以保留英文内部字段，但报告展示不得依赖英文键名。

## 验收标准

1. RAG 上传包中新增验证码能力边界材料，并在集成指南中列出。
2. 模块 3 规则明确“验证码不是自动阻塞项”，并包含验证码类型、指令、费用、准确率、人工兜底、合规授权判断。
3. 模块 2 到模块 5 的规则文件都写明本模块何时必须继续提问。
4. 系统提示词写明自动流转逻辑。
5. 模块 6 schema 和规则支持统一 HTML 报告。
6. 测试覆盖验证码边界、自动流转、统一报告输出契约。
7. 全量单元测试通过。
